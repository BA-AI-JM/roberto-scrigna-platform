## What This Is

A complete nutrition practice management platform for Roberto Scrigna, an Italian
sports nutritionist. Roberto manages clients, creates personalised meal plans with
clinically calculated macros, tracks client check-ins, sends invoices, and monitors
progress — all in Italian.

The platform has two sides:
- **Coach dashboard** — Roberto's operational cockpit (login at `/`)
- **Client portal** — read-only view for Roberto's clients (login at `/portal/login`)

## What the Next Developer Needs to Know

### The engine is the crown jewel
`src/engine/` is 100% pure TypeScript with zero side effects. No database, no network,
no React imports. 320 tests verify it against real clinical data. Don't touch it unless
you understand the nutrition science (spec v4.4 / v4.4.1).

### The plan bundle overloads daily_targets
`plan.daily_targets` is a JSONB column that stores the full engine output as
`plan_bundle` inside it. This is documented tech debt — should be its own column.
The comment at plan.ts:218 acknowledges this.

### Two auth systems
- Partner (Roberto): email/password via Supabase Auth → `/auth/callback`
- Client: magic link OTP via Supabase Auth → `/portal/auth/callback`
These are completely separate paths. Don't merge them.

### Inline styles vs Tailwind
The codebase has a split personality. Login/register/intake use Tailwind. Dashboard
pages use inline `style={{}}`. The shadcn components were installed but only the new
pages use them consistently. A full migration to Tailwind classes is v1.1 work.

### The PDF pipeline
`src/pdf/generator.ts` uses `@sparticuz/chromium` for Vercel and falls back to system
Chrome locally. The HTML renderer at `src/pdf/html-renderer.ts` has a professional
brand system (navy/gold) that the web UI should eventually adopt.

### Inngest requires post-deploy registration
The 8 Inngest functions are registered at `/api/inngest` but Inngest needs to know the
production URL. This is a manual step after first deployment — see DEPLOYMENT-GUIDE.md.
