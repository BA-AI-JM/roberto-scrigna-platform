# EXPERIENCE AUDIT — CODEX — 2026-07-19

**Evidence used:** code only (`src/**`) plus `docs/polish/baseline-sweep/manifest.json` for route inventory. No screenshots or live-app inspection were used, and no other audit/delight report was read. Counts below cover production `.ts`, `.tsx`, and `.css` under `src`, excluding tests, unless stated otherwise.

## Verdict: why it reads 4/10

The product reads as 4/10 because its intelligence is stronger than its presentation contract. The plan engine, check-in lifecycle, signature safeguards, PDFs, and granular queries encode serious capability, but the interface presents that capability as long stacks of neutral cards, tables, raw controls, and state-specific text rather than a coherent Roberto-led service. The root causes, in order, are: (1) the core coach loop is not choreographed—seven intake pages lead into a seven-section generation form and then a six-tab, 1,920-line review surface, while the generation “magic moment” is only a disabled button label; (2) identity fragments among teal app tokens, legacy navy/zinc inline styles, and a separate navy/gold PDF system; (3) state handling is locally competent but systemically inconsistent, especially on portal pages where query failures can look like empty data; (4) mobile and accessibility basics are partly present but not enforced by shared primitives; and (5) progress is reported as data, not interpreted as a motivating clinical story. The result feels like a capable internal admin tool attached to a serviceable patient portal, not one premium end-to-end nutrition experience.

## Micro findings (ordered by leverage)

### [P-COACH|M2/M4/M10|src/app/(dashboard)/plans/generate/page.tsx:572] The product’s magic is encoded as a configuration stack

**Evidence:** After client selection, the page conditionally reveals Obiettivo & deficit, Struttura del piano, Macro overrides, Source swap, Configurazione Pasti, and Impostazioni Avanzate—seven card-like sections including selection (`src/app/(dashboard)/plans/generate/page.tsx:572`; `src/app/(dashboard)/plans/generate/page.tsx:616`; `src/app/(dashboard)/plans/generate/page.tsx:820`; `src/app/(dashboard)/plans/generate/page.tsx:835`; `src/app/(dashboard)/plans/generate/page.tsx:865`; `src/app/(dashboard)/plans/generate/page.tsx:880`; `src/app/(dashboard)/plans/generate/page.tsx:982`). Optional controls are not collapsed. The live week preview query is driven directly by schedule, session, deficit, macro, and source-pin inputs with no debounce (`src/app/(dashboard)/plans/generate/page.tsx:374`; `src/app/(dashboard)/plans/generate/page.tsx:378`; `src/app/(dashboard)/plans/generate/page.tsx:396`); while fetching, the only feedback is “· aggiornamento…” and the old values remain in place (`src/app/(dashboard)/plans/generate/page.tsx:1326`; `src/app/(dashboard)/plans/generate/page.tsx:1357`).

**Deduction:** The screen mirrors engine inputs instead of the coach’s decision sequence. Required, inferred, and exceptional decisions have equal visual weight; expert confidence drops because it is unclear what must be reviewed versus what is safely automatic. Direct refetches also make rapid edits feel unsettled and can show stale-looking numbers without a clear pending treatment.

**Fix:** Turn the page into four progressive stages: (1) readiness and inherited client facts, (2) goal/safety decision, (3) week structure, (4) optional tuning. Put engine-owned defaults in a collapsed “Calcolato automaticamente” summary and reveal overrides only on intent. Debounce preview input by 300ms, preserve the last result with a visible updating veil, and label exactly which decision changed the output.

### [P-COACH|M4|src/app/(dashboard)/plans/generate/page.tsx:430] Generation has no wait experience and no reveal

**Evidence:** `plan.generate` routes immediately to review on success and stores only a raw error on failure (`src/app/(dashboard)/plans/generate/page.tsx:430`; `src/app/(dashboard)/plans/generate/page.tsx:432`; `src/app/(dashboard)/plans/generate/page.tsx:435`). Across production source there is no Framer Motion/motion import; motion is mostly short hover/toggle transitions, while global CSS only guarantees reduced-motion suppression (`src/app/globals.css:143`; `src/components/ui/button.tsx:8`).

**Deduction:** The highest-value computation feels like a blocking form submission. There is no evidence of work, no expectation setting, no reassurance that safety/periodisation logic is running, and no authored transition from inputs to a valuable artifact. This erases the product’s strongest possible signature moment.

**Fix:** Replace the button-only pending state with a route-safe generation panel that names real phases (validating safety floor, calculating training-day expenditure, balancing macros, composing meals, checking constraints), communicates that the coach can stay or leave, handles long-tail delay/retry, and culminates in an accessible plan reveal. Use deterministic phase copy rather than fake percentages; respect `prefers-reduced-motion`.

### [P-CLIENT|M3|src/app/portal/(protected)/dashboard/page.tsx:316] Portal query errors can masquerade as empty or partial content

**Evidence:** The dashboard launches six queries and passes only `.data` and `.isLoading` into most child views; none of their `.error`/`.isError` values are rendered (`src/app/portal/(protected)/dashboard/page.tsx:316`; `src/app/portal/(protected)/dashboard/page.tsx:366`; `src/app/portal/(protected)/dashboard/page.tsx:373`; `src/app/portal/(protected)/dashboard/page.tsx:385`). The plan page likewise gives `ActivePlanView` only data and loading (`src/app/portal/(protected)/plan/page.tsx:13`; `src/app/portal/(protected)/plan/page.tsx:23`), so an errored query with undefined data reaches the “Nessun piano attivo” branch (`src/components/portal/active-plan-view.tsx:106`). The diary query has no fetch-error branch (`src/app/portal/(protected)/diary/page.tsx:41`; `src/app/portal/(protected)/diary/page.tsx:94`), and Progress passes no errors for snapshots, profile, or documents (`src/app/portal/(protected)/progress/page.tsx:22`; `src/app/portal/(protected)/progress/page.tsx:47`; `src/app/portal/(protected)/progress/page.tsx:62`).

**Deduction:** A client can be told there is no plan, no diary data, no measurement, or no document when the real condition is network/server failure. The route-level error boundary does not solve handled React Query errors, because these pages continue rendering. This is the encoded form of “silent death.”

**Fix:** Standardize an async-view contract `{data, loading, error, refetch}` and prohibit undefined-as-empty. Render section-level retry states when one query fails, plus a page-level degraded-data notice when several fail. Log error codes without exposing internals. Only render empty states after a successful response explicitly returns empty.

## What 10/10 requires

10/10 does not require decorative motion or a wholesale rewrite of the clinical engine. It requires a product-direction decision and a focused experience rebuild around the two personas:

- **One identity:** decide the canonical color/type/voice system, then share it across app, portal, email, plan PDF, invoice PDF and signature artifacts. Migrate raw inline styling into governed primitives and artifact tokens.
- **One core workspace:** rebuild intake → generate → review → share as a persisted, recoverable journey with progressive disclosure, real wait/reveal states, exception-led review and explicit delivery confirmation. This is the largest scope and highest return.
- **One async-state contract:** loading, empty, error, degraded, pending, success and retry must mean the same thing everywhere. Portal query failures must never render as empty data.
- **A mobile client model:** design for 390px and the client’s immediate question—today’s food, next check-in, progress meaning, signed documents—not a compressed desktop information architecture. Enforce touch size and test long content.
- **Interpreted data:** preserve frozen clinical values, but add baseline/target/context, honest trend interpretation, meaningful-change thresholds and coach annotations. Data should answer “so what?” without manipulation.
